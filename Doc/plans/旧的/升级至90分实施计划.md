### 1.4 核心问题诊断

1. **架构断层**：前端（静态 HTML）与 API（serve）是两个独立服务，前端不调用 API，直接读取静态 JSON 文件，导致无法进行任何写操作
2. **后端能力闲置**：`auth.py`、`feedback.py`、`workflow.py`、`lifecycle.py`、`audit.py` 等模块已完整实现，但仅通过 CLI 暴露，未通过 API 或前端暴露
3. **协同能力为零**：尽管后端已实现评论/收藏/评分/审批/生命周期/审计等协同能力，前端完全无对应 UI，API 完全无对应端点
4. **无访问控制**：前端和 API 均无身份认证，所有功能公开访问
5. **无管理后台**：无任何管理功能界面（用户管理、审计日志、系统配置等）
6. **AI 能力缺失**：无问答机器人、无内容摘要、无去重合并

## 三、技术架构重构方案

### 3.1 当前架构问题

```
当前架构（断层）：
┌─────────────┐     ┌──────────────┐
│  前端静态页  │     │   API 服务   │
│  (读取JSON)  │     │  (8个端点)   │
└─────────────┘     └──────────────┘
       ↓                   ↓
┌─────────────────────────────────┐
│         静态 JSON 文件           │
└─────────────────────────────────┘
       ↓
┌─────────────────────────────────┐
│      后端模块（仅CLI调用）       │
│  auth/feedback/workflow/...     │
└─────────────────────────────────┘
```

**问题**：前端不调用 API，API 不调用后端协同模块，三者完全脱节。

### 3.2 目标架构（统一）

```
目标架构（统一）：
┌─────────────────────────────────────┐
│          统一 Web 服务               │
│  (HTTPServer + 静态文件 + API)      │
├─────────────────────────────────────┤
│  ┌───────────┐  ┌────────────────┐  │
│  │ 前端 SPA  │  │  REST API 层   │  │
│  │ (HTML/JS) │←→│ (40+ 端点)     │  │
│  └───────────┘  └────────────────┘  │
│                       ↓             │
│  ┌─────────────────────────────────┐│
│  │      业务逻辑层（已有模块）      ││
│  │ auth/feedback/workflow/lifecycle││
│  │ audit/analytics/backup/batch_ops││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘
```

### 3.3 架构重构核心任务

1. **统一服务入口**：将 `serve` 命令改造为同时提供 API 和静态前端文件的服务
2. **API 全覆盖**：将 CLI 的 47 个命令能力全部通过 REST API 暴露
3. **前端 SPA 化**：前端改为通过 fetch 调用 API，不再依赖静态 JSON
4. **认证中间件**：API 层添加 Token/Session 认证中间件

## 四、分阶段实施计划

### 阶段一：架构重构与 API 补齐（基础层）

**目标**：统一服务入口，补齐 API 端点，为前端提供完整的数据接口
**预计提升**：+15 分（系统集成拓展 +3，基础内容管理 +5，检索查询 +4，底层运维 +3）

#### 任务 1.1：统一 Web 服务入口

**文件**：`lib/api_server.py`（重构）、`lib/web_server.py`（新建）

**具体工作**：
1. 新建 `lib/web_server.py`，实现 `UnifiedWebServer` 类：
   - 继承 `ThreadingHTTPServer`（多线程并发，解决高并发问题）
   - 静态文件服务：提供 `views/` 目录下的 HTML/JS/CSS
   - API 路由：所有 `/api/*` 路径转发到 APIHandler
   - 其他路径：返回 `views/index.html`（SPA 路由）
2. 重构 `lib/api_server.py`：
   - 将 `APIRequestHandler` 改为支持 RESTful 风格
   - 添加路由分发机制（支持路径参数 `/api/atoms/{id}`）
   - 添加认证中间件（Token/Session）
3. 修改 `cmd_serve`：启动统一服务，同时提供前端和 API

**交付物**：
- `lib/web_server.py`：统一服务入口
- 重构后的 `lib/api_server.py`：支持路由分发和认证
- 修改后的 `cmd_serve`：一键启动完整服务

| 技术 | 用途 | 选型理由 |
|------|------|----------|
| Python 标准库 http.server | Web 服务 | 已有基础，无需引入 Flask/Django |
| ThreadingHTTPServer | 多线程并发 | 解决高并发问题 |
| 现有 lib/* 模块 | 业务逻辑 | 已完整实现，直接复用 |

**约束**：不引入 Flask/Django 等 Web 框架，保持零依赖特性（除可选依赖外）。

|------|------|------|
| 问答机器人 | 调用外部 LLM API | 支持 OpenAI/Anthropic/本地 Ollama |
| 内容摘要 | 调用外部 LLM API | 同上 |
| 语义检索 | 已有 ChromaDB + sentence-transformers | 直接复用 |

**约束**：LLM 调用为可选功能，不配置 API Key 时降级为纯检索模式。

## 七、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 前端复杂度激增 | 开发周期延长 | 分阶段交付，每阶段独立可用 |
| LLM API 依赖 | AI 功能不可用 | 设计降级模式，无 LLM 时返回检索结果 |
| 多线程并发安全 | 数据竞争 | 对写操作加锁，SQLite 启用 WAL 模式 |
| 浏览器兼容性 | 部分功能不可用 | 使用 ES6+，目标浏览器 Chrome 90+ |
| 静态文件缓存 | 更新不生效 | 版本号查询参数 `?v=timestamp` |
