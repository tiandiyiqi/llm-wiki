# PLAN-000 - llm-wiki 企业化改造项目总方案

> **创建时间**：2026-06-21
> **基准评分**：60.25/100
> **总投入预算**：80-128万元
> **总周期**：8-11个月

---

## 一、总体目标

将 llm-wiki 从个人知识库工具改造为企业级知识管理系统，同时保留个人 Skill 模式（file_mode）。

---

## 二、总体要求

### 2.1 核心要求

1. **双模式架构**
   - file_mode：保留 Skill 模式，Markdown 文件存储，Obsidian 兼容
   - db_mode：企业模式，PostgreSQL 存储，多用户协作

2. **企业级功能**
   - 高可用架构（PostgreSQL 主从）
   - SSO 集成（Casdoor）
   - 版本管理（Snapshot 方案）
   - 多级知识库（个人/部门/项目/公司）
   - 安全合规（审计日志不可篡改、数据加密）

3. **用户体验**
   - 搜索优化（高亮、联想、摘要）
   - OCR 能力（PaddleOCR）
   - 在线预览（PDF.js + KKFileView）
   - 移动端优化（响应式 + PWA）

### 2.2 约束条件

| 约束项 | 要求 |
|--------|------|
| 项目初衷保留 | file_mode 必须保留 Skill 特性 |
| 信创友好 | 优先国产开源方案 |
| 技术栈统一 | PostgreSQL 统一存储 |
| 渐进式改造 | 分阶段实施，每阶段可交付 |

## 六、关键技术决策

| 决策项 | 选定方案 | 理由 |
|--------|----------|------|
| 数据库 | PostgreSQL | 一库多能（JSONB + pgvector + RLS） |
| SSO | Casdoor | 国产开源、多协议、企业IM内置 |
| 版本管理 | Snapshot（PostgreSQL） | 无外部依赖、数据统一 |
| 容器化 | Docker Compose + GitHub Actions | 起步简单、可扩展 K8s |
| OCR | 云服务 API → PaddleOCR | 分阶段、信创友好 |
| 预览 | Open File Viewer + BaseMetas FileView | 前端轻量 + 后端强大 |
| 加密 | pgcrypto + 应用层 | 分层策略 |

**技术决策记录（2026-06-24）**：
- **数据加密方案**：分层加密策略
  - 数据库层：pgcrypto 扩展（透明加密，支持索引）
  - 应用层：cryptography 库（灵活控制，支持密钥轮换）
  - 加密字段：phone/email（用户隐私）、ip_address（审计日志）
  - HMAC 索引：支持加密字段的等值查询
- **文件预览方案**：采用混合模式
  - 主力：Open File Viewer（纯前端 SDK，零服务器依赖）
  - 备用：BaseMetas FileView（后端转换服务，支持 200+ 格式、OFD、CAD）
  - 本期策略：前端集成 Open File Viewer + 后端接口预留（返回提示信息），暂不部署 BaseMetas FileView
  - 选型理由：BaseMetas FileView 相比 KKFileView 更轻量（零 Office 依赖）、格式更全（含 OFD/CAD）、国产信创友好
