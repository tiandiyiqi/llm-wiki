## 一、数据模型设计

### 1.1 四维数据模型

**飞书方案**：
```
知识分类（单选）  → 技术文档/操作指南/FAQ
内容主体（长文本） → Markdown格式，最大1MB
关联标签（文本）   → 逗号分隔关键词，最多10个
元数据（组合字段） → 创建人/更新时间/版本号（自动生成）
```

**值得借鉴的理由**：

1. **结构清晰**：四个维度覆盖知识的"是什么、属于什么、关联什么、从哪里来"
2. **约束明确**：单选、必填、上限等约束保证数据质量
3. **自动生成**：元数据自动生成，降低人工录入成本
4. **便于检索**：分类和标签提供多维检索入口

**对 llm-wiki 的启示**：

```sql
-- 建议扩展（可选，非必需）
-- 当前已有：id, label, type, description, color, tags, in_degree, out_degree

-- 新增字段（锦上添花）
ALTER TABLE wiki_segments ADD COLUMN IF NOT EXISTS
  knowledge_category VARCHAR(50),   -- 补充分类维度
  metadata JSONB DEFAULT '{}'::jsonb; -- 扩展元数据

-- metadata 可包含：
-- {
--   "created_by": "user_id",
--   "created_at": "timestamp",
--   "version": 1,
--   "last_updated": "timestamp",
--   "source": "upload|api|manual",
--   "quality_score": 0.85
-- }
```

**实施建议**：
- 优先级：P2（非核心功能）
- 周期：1-2天
- 文件：`migrations/060_knowledge_metadata.sql`

## 二、建设流程标准化

### 2.1 四步法建设流程

**飞书方案**：
```
Step 1：创建/升级知识库
  → 新建"问答知识库"或将现有知识库发布为"问答知识库"

Step 2：投喂"粮草"（知识录入）
  → 多格式上传（Word/PDF/PPT/表格）
  → 自动解析、提取关键信息
  → 向量化处理建立语义索引

Step 3：全网分享
  → 知识库广场展示
  → 支持兑换码权限管理

Step 4：知识付费/变现
  → 兑换码解锁功能
  → 精准访问控制
```

**值得借鉴的理由**：

1. **流程清晰**：4步法降低认知负担，易于执行
2. **渐进式**：从创建到变现，逐步深化
3. **可量化**：每步都有明确的交付物
4. **用户友好**："投喂粮草"等比喻易于理解

**对 llm-wiki 的启示**：

```markdown
# llm-wiki 知识库建设SOP（建议新增）

## Step 1：创建知识库
- [ ] 选择知识库类型（个人/部门/项目/公司）
- [ ] 设置基本信息（名称、描述、权限）
- [ ] 配置层级关系（父知识库）
- [ ] 验证：知识库创建成功，列表可见

## Step 2：知识录入
- [ ] 选择录入方式：
  - file_mode：导入Markdown文件（Obsidian兼容）
  - db_mode：Web界面录入
- [ ] 自动处理：
  - Markdown解析（标题、链接、标签）
  - 向量化存储（pgvector）
  - 知识图谱生成（节点、边）
- [ ] 验证：知识条目入库，可搜索可检索

## Step 3：知识分享
- [ ] 内部成员邀请
- [ ] 权限分配（查看/编辑/管理）
- [ ] 知识库发现（搜索/推荐/分类）
- [ ] 验证：其他用户可访问知识库

## Step 4：知识运营
- [ ] 使用统计（访问量、搜索频次）
- [ ] 质量评分（用户反馈、AI评估）
- [ ] 版本管理（Snapshot回滚）
- [ ] 验证：运营数据可视化

## 附：验收标准
- 完成100条测试数据入库
- 检索准确率 ≥ 85%
- 用户满意度 ≥ 90%
```

**实施建议**：
- 优先级：P1（标准化流程）
- 周期：1天（文档编写）
- 文件：`Doc/SOP-knowledge-base-setup.md`

## 五、技术架构参考

### 5.1 全云架构方案

**飞书方案**：
```
AI引擎：DeepSeek V3模型（128K上下文窗口）
数据存储：飞书多维表格企业版（单表10万行）
中间件：飞书开放平台API（300次/分钟）
部署方式：全云架构（无需本地服务器）
```

**值得借鉴的理由**：

1. **容量明确**：单表10万行，为数据增长预留空间
2. **性能量化**：API调用限制300次/分钟，便于容量规划
3. **上下文充足**：128K窗口，支持长文档处理
4. **架构简洁**：全云架构降低运维复杂度

**对 llm-wiki 的启示**：

```markdown

## API性能规划
```

## 六、知识付费/变现机制

### 6.1 兑换码权限管理

**飞书方案**：
```
开启开关：进入知识库设置 → 找到「付费设置」 → 开启「加入知识库需兑换码」

生成兑换码：点击「兑换码奖励」→「生成兑换码」
  → 单次最多999个，总上限9999个
  → 可将兑换码作为奖励发给社区活跃用户
  → 或在第三方平台售卖，用户凭码入库

用户使用：
  → 3次免费试用（体验AI问答）
  → 输入16位兑换码解锁永久访问
```

**值得借鉴的理由**：

1. **商业模式完整**：从免费试用到付费解锁
2. **降低门槛**：3次免费试用降低决策成本
3. **灵活分发**：兑换码可作为奖励或售卖
4. **永久权益**：一次性兑换，长期受益

**对 llm-wiki 的启示**：

```markdown
# 知识库变现机制（长期规划，可选）

## 免费试用机制
- 搜索次数限制：每月10次免费搜索
- AI问答次数：每月3次免费问答
- 知识预览：允许预览前3段内容

## 付费解锁机制
- 个人版：¥99/年（无限制访问）
- 团队版：¥499/年（5人协作）
- 企业版：¥1999/年（无限制 + SSO）

## 兑换码管理
- 功能：
  - 批量生成兑换码（最多1000个）
  - 兑换码有效期（1年/永久）
  - 使用统计（已兑换/未兑换）
- 应用场景：
  - 社区活动奖励
  - 企业内部分发
  - 教育机构合作

## 技术实现
- 数据表：兑换码表（兑换码、状态、有效期、使用者）
- API：
  - POST /api/codes/generate（生成兑换码）
  - POST /api/codes/redeem（兑换解锁）
  - GET /api/codes/stats（统计查询）
- 权限：
  - 兑换后自动升级角色（guest → member）

## 优先级
- P3（长期规划，非当前阶段）
- 建议在阶段2完成后评估
```

- 优先级：P3（商业模式探索）
