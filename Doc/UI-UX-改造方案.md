# LLM Wiki UI/UX 改造可行性方案

> 配套文档：[`design.md`](./design.md)（设计系统单一事实来源）
> 目标方向：**专业克制风格（Linear/Notion/GitHub 系）+ 靛蓝主色 + 亮色为主暗色次要**
> 技术约束：**保持现有 Alpine.js + Tailwind 无编译技术栈**，不引入框架重写
> 版本：v1.0　|　日期：2026-06-26

---

## 一、方案总览

本方案在**不更换前端框架**的前提下，对现有 Web UI 做系统性 UI/UX 升级。改造分四个维度推进，按"风险低→高、收益快→慢"排成 4 个阶段，每阶段独立可交付、可回退。

| 阶段 | 主题 | 工期估算 | 风险 | 核心收益 |
|------|------|---------|------|---------|
| **P1** | 设计令牌地基 | 2-3 天 | 🟢 低 | 一处改色全局生效，为后续铺路 |
| **P2** | 组件视觉统一 | 3-5 天 | 🟢 低 | 14 工具 + 5 视图观感一致 |
| **P3** | 交互与信息架构 | 5-7 天 | 🟡 中 | 搜索/导航/图谱体验跃升 |
| **P4** | 响应式·可访问性·性能 | 4-6 天 | 🟡 中 | 移动端达标、WCAG AA、首屏提速 |

> 总计约 14-21 个工作日（单人）。各阶段可拆分给并行执行，详见第七节。

---

## 二、现状诊断（改造依据）

基于对现有项目的梳理，识别出以下可改进点：

### 🎨 视觉与设计语言
- **问题**：Tailwind 通过 CDN 实时配置，色值散落在各 HTML/JS 中，无统一令牌；缺少品牌色体系，观感偏"默认 Tailwind"。
- **影响**：换肤/暗色模式无从下手，14 个管理工具视觉细节不一致。

### 🧭 交互与信息架构
- **问题**：侧边栏经 PLAN-M-012 已治理（`nav-config.js` 单一来源），但**搜索体验、图谱交互、空/加载/错误状态**尚未系统设计；管理工具多为表格罗列，操作路径偏长。
- **影响**：高频用户效率未被充分优化。

### 📱 响应式与可访问性
- **问题**：PLAN-M-006 已做移动端 drawer 与 PWA，但**焦点环、键盘导航、对比度、触控目标**未系统校验。
- **影响**：可访问性未达 WCAG AA，键盘用户体验欠佳。

### ⚙️ 性能与工程质量
- **问题**：Tailwind CDN + 运行时 ES Module 动态 import（带 `?t=Date.now()` 破缓存），首屏依赖 CDN，无构建期 Tree-shaking。
- **影响**：首屏与弱网体验有优化空间。

## 三、P1 —— 设计令牌地基（🟢 低风险）

**目标**：把散落色值收敛为 CSS Variables + Tailwind 令牌，实现"一处改、全局变"。

### 改造内容
1. 新建 `/views/css/tokens.css`，按 `design.md` 第二节定义全部 `:root` 变量（色彩/字体/间距/圆角/阴影/动效）。
2. 将 Tailwind CDN 配置中的 `theme.extend` 指向 CSS 变量：
   ```js
   // index.html 内 tailwind.config
   theme: { extend: {
     colors: {
       primary: { 50:'var(--color-primary-50)', 600:'var(--color-primary-600)', /* ... */ },
       // 中性/语义色同理
     },
     borderRadius: { sm:'var(--radius-sm)', md:'var(--radius-md)' },
   }}
   ```
3. 暗色模式：`[data-theme="dark"]` 覆盖变量；顶栏主题切换按钮写入 `localStorage` + `prefers-color-scheme` 兜底。

### 交付物 & 验收
- `tokens.css`（亮+暗两套变量）
- 顶栏主题切换可用，刷新保持
- **验收**：任意改一个 `--color-primary-600`，全站主操作色同步变化

### 风险与回退
- 风险极低，纯增量；如异常，移除 `tokens.css` 引用即回退。

---

## 四、P2 —— 组件视觉统一（🟢 低风险）

**目标**：升级 `ui-components.js` 共享组件库视觉到 `design.md` 规范，让 14 个管理工具 + 5 个主视图自动同步。

### 改造内容
1. **重写组件样式**（结构不变，仅 class/样式按令牌调整）：
   - `createBadge` → 语义浅底深字（替换实底高饱和）
   - `createTable` → 行高 44px、表头 sticky、斑马纹、hover 高亮、操作列右对齐
   - `createModal` → 统一遮罩、尺寸档位、ESC 关闭
   - `createLoadingSpinner` → 主色描边 + 配套 **Skeleton 骨架屏**（新增）
2. **新增组件**（补齐缺口）：
   - `createButton(variant)` —— Primary/Secondary/Ghost 三级
   - `createEmptyState(cfg)` —— 统一空状态
   - `createInput(cfg)` —— 统一聚焦环
3. **侧边栏 NavItem**：选中态加左侧 3px 主色条 + `bg-primary-50`，侧边栏加宽至 240px。

### 交付物 & 验收
- 升级版 `ui-components.js`，新增 3 个组件
- 组件预览页 `/views/admin/_components.html`（自查用）
- **验收**：抽查 5 个管理工具，按钮/徽章/表格/弹窗视觉与规范一致

### 风险与回退
- 仅改样式不改接口，向后兼容；逐组件灰度，问题可单组件回退。

---

## 五、P3 —— 交互与信息架构（🟡 中风险）

**目标**：优化高频路径，提升搜索、导航、图谱、状态反馈体验。

### 改造内容
1. **全局搜索升级**（最高收益）
   - 引入 **命令面板（⌘K / Ctrl+K）**：全局唤起，聚合搜索原子 + 跳转视图 + 快捷操作。
   - 搜索结果支持分组（标题/标签/正文命中）、键盘上下选择、高亮命中。
   - 复用现有 `/api/search`、`/api/suggest`，前端仅做交互层。
2. **导航与面包屑**
   - 主视图 + 管理工具加面包屑，明确"我在哪"。
   - 侧边栏分组（内容操作 / 系统管理）加可折叠分区与图标。
3. **图谱（graph）交互**
   - 增加节点详情侧滑面板（点击节点不跳页，侧滑看详情+关系）。
   - 布局预置下拉 + 按类型筛选 + 搜索定位高亮。
4. **状态体系标准化**
   - 全站统一"加载（Skeleton）/ 空（EmptyState）/ 错误（可重试）"三态，替换裸 spinner 与空白页。
5. **编辑体验**
   - `edit` 视图：自动保存提示、未保存离开拦截、Markdown 实时预览分屏。

### 交付物 & 验收
- 命令面板组件 `/views/components/command-palette.js`
- 图谱详情侧滑面板
- 三态标准化覆盖全部列表/表格视图
- **验收**：⌘K 可全局搜索跳转；任一列表在加载/空/错误下均有规范态

### 风险与回退
- 中风险：命令面板与图谱交互为新增逻辑，需充分测试键盘/焦点。
- 回退：命令面板为独立模块，可整体下线；图谱面板可降级回原跳转。

## 六、P4 —— 响应式·可访问性·性能（🟡 中风险）

**目标**：移动端体验达标、满足 WCAG 2.1 AA、首屏与弱网提速。

### 改造内容
1. **响应式校验**
   - 桌面侧边栏 ↔ 移动 drawer 数据已同源（PLAN-M-012），本阶段校验断点、表格→卡片堆叠、触控目标 ≥ 44px。
2. **可访问性（A11y）**
   - 全局焦点环（`:focus-visible`），键盘可达所有交互。
   - 补 `aria-label` / `role`，表单关联 `<label>`。
   - 对比度审计（靛蓝主色已达 7.0:1）。
   - 尊重 `prefers-reduced-motion`。
3. **性能（按收益排序，可选）**
   - **首屏 CSS**：评估将 Tailwind 从 CDN 改为**构建期生成**（仅此项需引入轻量构建步骤，如 Tailwind CLI，**不改框架**），产出 purge 后的静态 CSS，首屏不依赖 CDN。
   - **模块加载**：用版本号替代 `?t=Date.now()` 破缓存，启用浏览器缓存；管理工具模块按需预取（prefetch）。
   - **图谱**：大数据量分批渲染 / 懒加载。

### 交付物 & 验收
- A11y 审计报告（键盘走查 + 对比度）
- 响应式断点截图对比（移动/平板/桌面）
- 性能基线对比（首屏、CDN 依赖数）
- **验收**：键盘可完成核心流程；Lighthouse A11y ≥ 95

### 风险与回退
- "Tailwind 构建化"是本方案唯一触及工程链的改动，**设为可选项**：若希望严格保持零构建，可保留 CDN，仅做缓存与按需加载优化。

## 七、执行建议与并行编排

### 依赖关系
```
P1（令牌地基）──┬─→ P2（组件统一）──→ P3（交互架构）
                └─────────────────────→ P4（响应式/A11y/性能）
```
- **P1 是地基，必须先行**；P2 依赖 P1。
- P3 与 P4 可在 P2 完成后**并行**（触及不同关注点）。

### 推荐落地顺序
1. 先做 **P1 + P2**（约 1 周）：风险最低、观感提升最直观，可快速验证设计方向。
2. 再 **P3 / P4 并行**（约 1.5 周）：体验深化与工程优化。

### 关键原则
- 每阶段独立提交、独立可回退（遵循项目 PLAN-M 系列的增量交付习惯）。
- 所有改动收敛到 `design.md` + `tokens.css` + `ui-components.js` + `nav-config.js` 四个单一来源，杜绝散落硬编码。
- 改造期间保持现有 API 契约不变，纯前端演进。

## 八、风险登记表

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| Tailwind CDN 配置与 CSS 变量映射兼容性 | 🟢 低 | P1 先在单页验证，确认后全量 |
| 14 工具样式升级遗漏个别视图 | 🟡 中 | 建组件预览页逐一对照；灰度回退 |
| 命令面板/图谱新交互的键盘与焦点缺陷 | 🟡 中 | 专项 A11y 走查，模块化可下线 |
| Tailwind 构建化引入工程链复杂度 | 🟡 中 | 设为 P4 可选项，可保留 CDN |
| 暗色模式对比度不达标 | 🟢 低 | 令牌层统一调整，集中校验 |
